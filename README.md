# Relay

ระบบส่งไฟล์ข้ามอุปกรณ์แบบ Peer-to-Peer (P2P) โดยตรงผ่านเว็บเบราว์เซอร์ โดยถอดแบบความสะดวกจากบริการส่งไฟล์ชั้นนำ แต่เน้นการสื่อสารตรงระหว่างอุปกรณ์คู่สนทนา (Direct Connection) โดยไฟล์และข้อมูลดิบของผู้ใช้งานจะไม่ผ่านเซิร์ฟเวอร์แม่ข่ายแม้แต่ไบต์เดียวเพื่อความปลอดภัยและความเป็นส่วนตัวสูงสุด

เข้าใช้งานหน้าเว็บระบบได้ที่: [https://relay.benyapol.online/](https://relay.benyapol.com/)

---

## คุณสมบัติเด่นของระบบ

- การเชื่อมต่อแบบ Peer-to-Peer (P2P) แท้จริง: ไฟล์ถูกส่งตรงจากเครื่องผู้ส่งถึงผู้รับผ่านโปรโตคอล WebRTC DataChannel เซิร์ฟเวอร์ทำหน้าที่เป็นเพียงตัวช่วยส่งผ่านข้อมูลจับคู่ (Signaling) ในตอนแรกเท่านั้น
- ความปลอดภัยสูงสุดโดยธรรมชาติ: ข้อมูลในท่อ WebRTC ได้รับการเข้ารหัสตั้งแต้ต้นทางถึงปลายทาง (End-to-End Encryption - E2EE) ด้วยมาตรฐาน DTLS (Datagram Transport Layer Security) และ SRTP
- ส่งไฟล์ขนาดใหญ่ได้ไม่จำกัด: เนื่องจากไม่ต้องอัปโหลดขึ้นฝากที่เซิร์ฟเวอร์ จึงไม่มีการจำกัดขนาดไฟล์ส่ง
- การจัดการหน่วยความจำอัจฉริยะ (Flow Control): ควบคุมอัตราการส่งข้อมูลด้วยระบบตรวจวัดความจุของ Buffer ฝั่งรับ (ตั้งเพดานหยุดส่งชั่วคราวเมื่อ Buffer สูงเกิน 1MB และส่งต่อเมื่อเคลียร์เหลือต่ำกว่า 256KB) ป้องกันปัญหาหน่วยความจำเครื่องบวม (RAM Overflow) แม้ส่งไฟล์ระดับกิกะไบต์
- เทคโนโลยี Stream-to-Disk: เขียนไฟล์ลงบนหน่วยบันทึกข้อมูลของเครื่องรับโดยตรงแบบเรียลไทม์ผ่าน File System Access API บนเบราว์เซอร์ที่รองรับ และสลับไปใช้โหมดจัดเก็บบน Memory Blob เป็นระบบสำรองสำหรับเบราว์เซอร์หรืออุปกรณ์ทั่วไปที่ไม่สนับสนุน
- สะดวกรวดเร็วโดยไม่ต้องลงทะเบียน: เชื่อมโยงอุปกรณ์ด้วยรหัสตัวเลข 6 หลักแบบใช้ครั้งเดียว
- ระบบรายงานสถิติแบบเรียลไทม์: รวบรวมตัวเลขการประมวลผลการส่งข้อมูลรวมทั่วทั้งเซิร์ฟเวอร์แบบไม่ระบุตัวตน แสดงผลผ่านแดชบอร์ดด้านล่างของหน้าเว็บ

## สถาปัตยกรรมระบบและเทคโนโลยีหลัก

ระบบถูกออกแบบมาโดยเน้นประสิทธิภาพและความง่ายในการดูแลรักษา โดยพึ่งพาโมดูลภายนอกให้น้อยที่สุด (Lightweight Design)

| ส่วนประกอบ | เทคโนโลยีที่เลือกใช้ | บทบาทหน้าที่ในการประมวลผล |
|---|---|---|
| ส่วนหน้าบ้าน (Frontend) | Vanilla HTML5 / Vanilla CSS3 / Vanilla JS | พัฒนาในรูปแบบ Single Page Application (SPA) ทำงานได้โดยตรงโดยไม่ต้องมีกระบวนการ Build step หรือโหลดเฟรมเวิร์กขนาดใหญ่ ช่วยยกระดับความเร็วในการแสดงผล |
| ระบบส่งสัญญาณ (Signaling Server) | Node.js (Express) + Socket.io | ทำการรับคำขอจากเบราว์เซอร์เพื่อจัดสรรห้องจับคู่อุปกรณ์ ส่งต่อข้อความ SDP (Session Description Protocol) และเส้นทางไอพี (ICE Candidates) เมื่อส่งสำเร็จจะยกเลิกการจับคู่เพื่อความปลอดภัย |
| ฐานข้อมูลสถิติ (Database) | SQLite (`node:sqlite`) | ใช้โปรแกรมฐานข้อมูลในตัวที่มีมาให้ตั้งแต่ Node.js 22 ขึ้นไป (DatabaseSync) จัดเก็บเพียงตัวเลขยอดรวมสถิติความสำเร็จในการให้บริการ โดยไม่มีการบันทึกประวัติการส่ง ข้อมูลชื่อไฟล์ หรือข้อมูลตัวตนผู้ใช้งานใดๆ ทั้งสิ้น |
| กลไกส่งผ่านข้อมูล | WebRTC (DataChannel) | สร้างช่องทางการสื่อสารสองทิศทางแบบ P2P ผ่านเครือข่ายอินเทอร์เน็ตโดยตรง |
| เว็บเซิร์ฟเวอร์หลัก (Web Server) | Apache (Reverse Proxy) | ใช้เป็นจุดรับทราฟฟิก HTTPS และคัดแยกส่งต่อไปยัง Node.js ภายในเครื่องผ่านโมดูล mod_proxy และ mod_wstunnel |

## กลไกการทำงานเบื้องหลัง (Under the Hood Flow)

กระบวนการจับคู่และส่งผ่านไฟล์ประกอบไปด้วย 5 ขั้นตอนหลัก:

1. การเปิดห้องและจัดสรรรหัสผ่าน (Room Creation)
   - ผู้ส่งเลือกไฟล์ที่ต้องการจะส่งบนหน้าเว็บเบราว์เซอร์
   - สคริปต์ส่งสัญญาณร้องขอรหัสผ่านไปยัง Signaling Server
   - Server สุ่มสร้างรหัส 6 หลักที่ยังไม่ถูกใช้งานในหน่วยความจำ และผูกสถานะผู้ส่งไว้กับรหัสนี้

2. การร่วมห้องและจับคู่คู่สาย (Peer Matching)
   - ผู้รับนำรหัสตัวเลข 6 หลักมากรอกและกดยืนยัน
   - Server ตรวจสอบรหัส หากพบห้องว่างและมีผู้ส่งเปิดรออยู่ จะอนุญาตให้เชื่อมโยงพร้อมส่งสถานะตอบกลับให้ทั้งสองฝั่งเริ่มทำงาน

3. การแลกเปลี่ยนข้อมูลเชื่อมโยง (WebRTC Handshake)
   - ฝั่งผู้ส่งสร้าง SDP Offer อธิบายความสามารถเครือข่ายของตนเองและส่งผ่าน Socket.io ไปหาผู้รับ
   - ผู้รับนำไปตั้งค่าและสร้าง SDP Answer ส่งกลับไปยังผู้ส่ง
   - ทั้งสองฝั่งเริ่มส่งข้อมูล ICE Candidates แลกเปลี่ยนหาเส้นทางที่เหมาะสมที่สุดในการคุยตรงหากัน (อาจใช้วิธีเจาะช่องทาง NAT)

4. การสตรีมข้อมูลและควบคุมความเร็ว (Data Streaming & Flow Control)
   - เมื่อท่อข้อมูล DataChannel เปิดใช้งานสำเร็จ ฝั่งผู้ส่งเริ่มอ่านไฟล์ผ่าน API ของเบราว์เซอร์ โดยแบ่งเนื้อไฟล์ออกเป็นขนาดชิ้นละ 64KB (Chunks) และส่งผ่านท่อไปทีละชิ้น
   - ผู้รับจะรับชิ้นข้อมูลและเขียนลงดิสก์โดยตรงผ่าน File System Access API
   - ในขณะส่ง หากท่อส่งข้อมูลเริ่มมีบัฟเฟอร์ค้างเกิน 1MB ฝั่งส่งจะระงับการสตรีมชั่วคราว รอสัญญาณแจ้งเตือนค่าบัฟเฟอร์ลดลงต่ำกว่า 256KB จากท่อส่งจึงเริ่มทำงานต่อ เพื่อควบคุมความสมดุลของการรับส่งข้อมูลและเซฟการใช้แรม

5. การสิ้นสุดกระบวนการส่งและนับสถิติ (Finalization)
   - เมื่อส่งข้อมูลครบทุกชิ้น ฝั่งผู้ส่งส่งสัญญาณเตือนการปิดไฟล์ และรอรับสัญญาณยอมรับบันทึกสำเร็จ (ACK) จากผู้รับ
   - ฝั่งส่งส่งตัวเลขปริมาณไบต์และจำนวนไฟล์ไปให้เซิร์ฟเวอร์เพื่อบันทึกลงฐานข้อมูล SQLite
   - ห้องและรายละเอียดการเชื่อมต่อ Socket จะถูกทำลายลงทันทีเพื่อความปลอดภัย

```
│       ├── webrtc.js           # ฟังก์ชันการจัดสรรท่อ P2P แบ่งส่วนข้อมูลไหลเข้าดิสก์ และควบคุมความเร็วส่ง

└── test_runner.js              # ชุดทดสอบอัตโนมัติสำหรับตรวจสอบความถูกต้องของการประมวลผลฐานข้อมูล
```

## วิธีการติดตั้งเพื่อพัฒนาในเครื่องคอมพิวเตอร์ของคุณ (Local Setup)

### ข้อมูลระบบที่ต้องการ
- Node.js เวอร์ชัน 22 ขึ้นไป (ต้องการเพื่อใช้งานโมดูลฐานข้อมูล SQLite ในตัวที่ไม่มีความจำเป็นต้องคอมไพล์ใหม่)

### ขั้นตอนการรันระบบภายในเครื่อง

1. เปลี่ยนที่อยู่ของ Command Line ไปยังโฟลเดอร์ที่จัดเก็บไฟล์โปรเจกต์:
   ```bash
   cd C:\xampp\htdocs\relay
   ```

2. ติดตั้งไลบรารีและ Dependencies ที่เกี่ยวข้อง:
   ```bash
   npm install
   ```

3. เริ่มเปิดรันแอปพลิเคชัน:
   - รันในโหมดปกติเพื่อเปิดเซิร์ฟเวอร์ใช้งาน:
     ```bash
     npm start
     ```
   - รันในโหมดนักพัฒนาเพื่อตรวจจับการเปลี่ยนแปลงและอัปเดตไฟล์อัตโนมัติ (Watch Mode):
     ```bash
     npm run dev
     ```

4. ทดลองเข้าใช้งานผ่านเบราว์เซอร์:
   - ลิงก์เข้าหน้าเว็บทดสอบ: http://localhost:3000
   - ลิงก์ตรวจสอบค่าสถิติระบบ: http://localhost:3000/api/stats

## การทดสอบความถูกต้องของโปรแกรม (System Testing)

โครงการนี้มีไฟล์ทดสอบความถูกต้องของการประมวลผลการจัดเก็บสถิติและการดักกรองข้อมูลแปลกปลอม (`test_runner.js`) เพื่อทดสอบความสมบูรณ์และป้องกันข้อมูลผิดพลาดเข้าสู่ฐานข้อมูล SQLite สามารถรันการตรวจสอบได้โดยพิมพ์คำสั่ง:

```bash
node test_runner.js
```

ตัวโปรแกรมทดสอบจะทดสอบกระบวนการบันทึกข้อมูลปกติและการดักสกัดข้อมูลขยะ (เช่น ตัวเลขติดลบ, Infinity, ข้อความที่ไม่ใช่ตัวเลข หรือตัวเลขส่งไฟล์ที่สูงผิดปกติเกินขีดจำกัดความปลอดภัยของระบบ) และจะรายงานผลลัพธ์การทำงานผ่านหน้าจอคอมมานด์ไลน์

## การติดตั้งสำหรับใช้งานจริงร่วมกับเซิร์ฟเวอร์ Apache (Production Setup)

หากต้องการทำโดเมนใช้งานจริงอย่างปลอดภัยภายใต้ HTTPS (เช่น https://relay.benyapol.online/) คุณสามารถตั้งค่า Apache Web Server เป็นตัวผ่าน (Reverse Proxy) เข้าหาพอร์ต 3000 ของ Node.js ได้ตามตัวอย่างขั้นตอนดังนี้:

1. เปิดไฟล์ตั้งค่าหลักของ Apache (`httpd.conf`) และตรวจสอบว่าโมดูลต่อไปนี้เปิดทำงานแล้ว (ไม่มีสัญลักษณ์ `#` นำหน้า):
   ```apache
   LoadModule proxy_module modules/mod_proxy.so
   LoadModule proxy_http_module modules/mod_proxy_http.so
   LoadModule proxy_wstunnel_module modules/mod_proxy_wstunnel.so
   LoadModule rewrite_module modules/mod_rewrite.so
   LoadModule ssl_module modules/mod_ssl.so
   LoadModule headers_module modules/mod_headers.so
   ```

2. เชื่อมโยงไฟล์คอนฟิกตัวอย่างเข้าไปทำงานโดยระบุคำสั่งไว้ที่จุดท้ายสุดของไฟล์ `httpd.conf`:
   ```apache
   Include "C:/xampp/htdocs/relay/config/apache-relay.conf"
   ```

3. เพิ่มข้อมูลแผนที่ไอพีชี้โดเมนภายในระบบโดยเปิดไฟล์ hosts ของเครื่องวินโดวส์ที่ `C:\Windows\System32\drivers\etc\hosts` และเขียนเพิ่มบรรทัดดังต่อไปนี้:
   ```hosts
   127.0.0.1  relay.benyapol.online
   ```

4. ทำการรีสตาร์ทแอปพลิเคชัน Apache และดำเนินการสั่งรันเซิร์ฟเวอร์หลักโดยเรียกใช้คำสั่ง `npm start` ในโฟลเดอร์โปรเจกต์

## การกำหนดค่าคงที่ผ่านตัวแปรสภาพแวดล้อม (Environment Variables)

ระบบรองรับการกำหนดพฤติกรรมการทำงานของบริการผ่านตัวแปรสภาพแวดล้อมดังนี้:

| ตัวแปร | ค่าเริ่มต้น | รายละเอียดคำอธิบาย |
|---|---|---|
| PORT | 3000 | ระบุหมายเลขพอร์ตที่ต้องการเปิดบริการสำหรับ Node.js |
| NODE_ENV | (ไม่มี) | หากระบุค่าเป็น `production` ระบบจะเปิดระบบส่งเก็บแคชเนื้อหาหน้าเว็บเพื่อความรวดเร็วในการส่งข้อมูล |
| ALLOWED_ORIGINS | * | กำหนดโดเมนที่สามารถเข้าถึงระบบข้ามเซิร์ฟเวอร์ผ่าน CORS ได้ตามต้องการ (แยกค่าด้วย `,` สำหรับหลายโดเมน) |

## ข้อมูลและข้อจำกัดสำคัญเกี่ยวกับการใช้งาน

- ความร่วมมือของระบบเขียนข้อมูลลงไดรฟ์โดยตรง:
  ฟังก์ชัน File System Access API สำหรับสตรีมข้อมูลลงดิสก์โดยตรงจะทำงานได้ดีที่สุดบน Google Chrome และ Microsoft Edge เวอร์ชันคอมพิวเตอร์ ส่วน Safari และ Firefox รวมถึงอุปกรณ์เคลื่อนที่ทั่วไปจะสลับไปบันทึกผ่านหน่วยความจำชั่วคราวก่อนดาวน์โหลด ซึ่งควรจำกัดการส่งเฉพาะไฟล์ขนาดไม่ใหญ่เกินกว่าแรมของอุปกรณ์นั้นๆ
- การใช้งานผ่านระบบอินเทอร์เน็ตที่อยู่หลังไฟร์วอลล์หนาแน่น:
  ค่าเริ่มต้นของระบบใช้บริการ STUN Server สาธารณะของ Google ในกรณีที่หน้าเว็บทั้งสองฝั่งใช้เน็ตขององค์กรที่มีการสกัดกั้นทราฟฟิก P2P อย่างเข้มงวด อาจทำให้การคุยแบบ Peer-to-Peer ล้มเหลว วิธีแก้คือการนำเครือข่าย TURN Server มาใส่เพิ่มใน `RTC_CONFIG` ในสคริปต์ `public/js/webrtc.js` เพื่อทำหน้าที่ส่งผ่านทราฟฟิกข้อมูลที่เข้ารหัสในกรณีที่ไม่สามารถคุยแบบตรงได้
- การดักล้างข้อมูลความปลอดภัย:
  สคริปต์ฝั่งเบราว์เซอร์มีตัวล้างขยะ (Cleanup Hooks) หากฝั่งรับหรือส่งกดออกจากหน้าเว็บหรือตัดการเชื่อมต่อระหว่างการส่ง ระบบจะยกเลิกการเขียนไฟล์และเคลียร์แรมทิ้งทันที
